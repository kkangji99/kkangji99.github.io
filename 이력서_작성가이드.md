# 백엔드 개발자 이력서 작성 가이드

## 전체 구조 (섹션 순서)

1. **소개 (Intro)** — 사진, 한 줄 소개, 연락처(이메일/GitHub/블로그), 간단한 지향점
2. **기술 스택** — 언어/프레임워크/DB/인프라를 카테고리별로, "얼마나 능숙한지"보다 "어디에 왜 썼는지" 위주
3. **프로젝트 (핵심)** — 프로젝트당 1페이지 분량, 아래 상세 참고
4. **경력/경험** — 인턴, 사이드 프로젝트, 오픈소스 기여 등
5. **트러블슈팅/성과 하이라이트** — 프로젝트 섹션과 별도로 임팩트 있는 것 2~3개 요약
6. **블로그/학습 기록** (있다면)
7. **Contact**

## 프로젝트 섹션 — 가장 중요한 부분

백엔드는 UI가 없으니 "어떻게 보이는가"가 아니라 "어떻게 설계했는가"로 승부해야 한다. 프로젝트마다 아래 틀을 반복한다.

- **개요**: 무엇을 만들었나, 기간, 팀 규모(1인/N인), 본인 역할
- **기술 스택**: 사용 이유를 한 줄씩 (예: "Redis — 세션 조회 응답시간 단축 목적")
- **아키텍처 다이어그램**: 시스템 구조도 이미지 첨부 (draw.io, excalidraw 등)
- **ERD / API 명세**: 핵심 테이블 구조, 주요 엔드포인트 요약
- **트러블슈팅 (STAR 방식)**: 상황 → 문제 → 시도한 해결책 → 결과(수치화)
  - 예: "동시접속 500명 시 응답지연 → N+1 쿼리 원인 파악 → 쿼리 최적화·인덱스 추가 → 응답시간 1.2s→180ms"
- **결과/배운 점**: 성능 수치, 트래픽 처리량, 배포 후 지표 등
- **링크**: GitHub 레포, 배포 URL(가능하면), 발표자료

## 작성 원칙

- **수치화**: "성능 개선했음" 대신 "응답속도 70% 감소" 처럼 구체적 숫자로 표현
- **본인 기여 명확히**: 팀 프로젝트면 "내가 담당한 부분"을 별도로 표시
- **최신순 정렬**, 프로젝트는 2~4개로 압축 (양보다 질)
- **코드 스니펫은 최소화**, 대신 GitHub 링크로 유도
- 사내식당·워라밸 좋은 기업(규모 있는 조직)을 목표로 한다면, 협업 방식(코드리뷰, 컨벤션, Git 브랜치 전략) 언급이 플러스 요인

## 참고

- 이력서 원본(개인정보 포함)은 저장소에 커밋하지 않는다. 이 문서는 작성 기준만 담은 가이드다.
- 포트폴리오 사이트(`index.html`)의 Experience 섹션은 이 가이드의 "프로젝트" 틀(개요-기술스택-트러블슈팅-성과-링크)을 참고하여 구성한다.
